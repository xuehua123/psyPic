# 19. 图片工作台改造蓝图

## 目标

本文件把 PsyPic 从“AI 生图站”进一步收敛为“项目级图片工作台”。它不是推翻现有 V1，而是在现有创作台、项目侧栏、版本流、素材库、任务队列和社区能力上，补一层长期产品架构：

> PsyPic 的主形态应是 Project Workspace + Panel Shell + Image Job Runtime + Versioned Asset System，而不是单张图片预览器或单页表单生成器。

## 可借鉴的工作台原则

### 1. Workspace-first

图片不是孤立结果，而是项目中的资产。后续文档统一把以下对象纳入同一个 workspace mental model：

- project：商业项目、客户项目、活动项目或个人创作项目。
- conversation/session：围绕一个目标持续生成和编辑的创作线索。
- version node：每次生成、编辑、同款、批量展开产生的可回溯节点。
- asset：图片、参考图、mask、导出文件和社区发布作品。
- task/job：生成、上传、批量、导出、审核、索引等后台任务。

### 2. Panel Shell，而不是大表单页面

桌面端默认工作区应稳定为：

```text
Project Sidebar      Main Canvas / Transcript        Inspector
项目、会话、素材       画廊、版本、对话、批量          参数、参考图、模板、历史、任务详情

Bottom / Dock
任务队列、批量状态、导出、运行日志
```

移动端不复制三栏，而是保留主创作流，把 Project Sidebar 和 Inspector 收进 Sheet / Drawer。

### 3. Job Runtime 是一等能力

图片生成不是单次 HTTP 请求体验，而是 job lifecycle：

- queued / running / partial / succeeded / failed / canceled / timed_out。
- 支持取消、重试、失败恢复、并发限制、超时归档。
- UI 需要有可观察的任务状态，而不是只靠按钮 loading。
- 批量生成、ZIP 导出、素材同步、社区发布审核都应复用 job mental model。

### 4. Versioned Asset System

每张图都要能回答：

- 从哪个 prompt / template / reference / mask 来？
- 属于哪个 project / session / branch？
- 是否是某个旧节点的 non-destructive branch？
- 是否已发布、收藏、打标签、加入相册、导出？
- 能否恢复参数、继续编辑、作为参考图、生成同款？

### 5. Search / Command Surface

后续不只做文件名或素材搜索。工作台搜索应覆盖：

- project、session、version node。
- board document、board export。
- prompt、template、scene、tag、尺寸、格式、usage、request id。
- asset、community work、album、batch task。
- 常用命令：新建项目、导入素材、批量导出、打开设置、从当前图分叉。

### 6. Multi-engine / Multi-provider Boundary

当前默认是 Sub2API + `gpt-image-2`。但文档和代码边界应避免把所有能力写死到一个 provider：

- generation provider：Sub2API / OpenAI Images API / Responses image tool。
- image utility provider：未来 OCR、背景移除、超分、压缩、审核。
- storage provider：local / S3 / R2 / MinIO。
- queue provider：in-memory / Redis / DB-backed worker。

### 7. Board Mode 是视觉编排层

“可画式 Board Mode”应作为工作台的一等模式，而不是结果卡上的小工具。它承载：

- 多张素材和历史结果的拼接、缩放、旋转、层级排序。
- 手绘标注、构图草图、局部编辑 mask 和文字 brief。
- 扁平化 reference PNG 导出，进入 `/api/images/edits`。
- alpha mask PNG 导出，进入 edits 的 `mask` 字段。
- board snapshot 和 export asset 进入 version node，支持恢复、分叉和搜索。

完整方案见 [20-多素材多图层画布（BoardMode）改造方案.md](./20-多素材多图层画布（BoardMode）改造方案.md)。

## 当前代码已有基础

当前仓库已经有多项工作台化基础，不需要从零开始：

- `components/creator/studio/ProjectSidebar.tsx`：项目侧栏。
- `components/creator/studio/ProjectCard.tsx`：平铺折叠项目卡。
- `lib/creator/projects-store.ts` / `use-projects.ts`：本地项目模型。
- `lib/creator/version-graph.ts`：版本节点和分支语义。
- `components/creator/studio/VersionStreamSection.tsx`：版本流展示。
- `components/creator/studio/inspector/*`：Inspector 分区。
- `server/services/image-job-queue-service.ts`：任务队列基础。
- `app/api/history`、`app/api/library`、`app/api/tasks`：历史、素材、任务 API。

文档改造的重点不是发明新方向，而是让产品、架构、API、数据模型和验收口径与这些代码事实对齐。

## 后续能力优先级

### P0：现有工作台事实文档化

- 文档统一使用 project / session / version node / asset / job。
- API 文档说明哪些能力现在是 local-first，哪些未来上服务端。
- 验收清单覆盖 ProjectSidebar、Inspector、VersionStream、Job Queue。

### P1：服务端化边界

- 本地 project store 可以继续保留，但文档预留 `WorkbenchProject` 服务端模型。
- Version graph 第一阶段 local-first，正式版可迁移到 DB。
- 搜索第一阶段局部筛选，正式版再接索引。
- BoardDocument 第一阶段也应 local-first，先验证画布交互、导出和版本快照，再服务端化。

### P2：扩展 provider

- 不急着接 OCR / background removal / upscaler。
- 但架构文档要保留 image utility provider 位置，避免将来挤进生成接口。
- Board export 只产出标准 image asset / mask asset，不把 Konva 或 canvas 内部结构泄露到 provider 层。

## 不改造的边界

- 不把 PsyPic 改成桌面 Tauri 应用；当前技术栈仍是 Next.js + BFF。
- 不照搬 AI code editor 的 chat thread 复杂度；图片工作台只吸收 workspace、panel、runtime、search、version 的产品结构。
- 不把旧 implementation plan 改写成新计划；历史计划保留为交付记录。
- 不在文档层面新增未确认付费、团队权限、复杂多人协作承诺。
