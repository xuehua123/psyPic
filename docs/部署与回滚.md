# 部署与回滚

## 目标

本文用于 PsyPic V1.0 生产封板前的部署检查、健康验证和回滚演练。生产服务器只允许拉取预构建产物、重启服务、健康检查和回滚，不在生产业务服务器构建 Docker 镜像。

## 环境变量

必需：

- `APP_URL`
- `DATABASE_URL`
- `SESSION_SECRET`
- `KEY_ENCRYPTION_SECRET`
- `SUB2API_DEFAULT_BASE_URL`
- `SUB2API_TIMEOUT_MS=300000`
- `PSYPIC_MAX_IMAGE_N`
- `PSYPIC_MAX_UPLOAD_MB`
- `PSYPIC_MAX_SIZE_TIER`
- `PSYPIC_PUBLIC_PUBLISH_ENABLED`
- `PSYPIC_COMMUNITY_ENABLED`
- `PSYPIC_STREAM_ENABLED`
- `PSYPIC_RUNTIME_SETTINGS_STORE=file|database`
- `PSYPIC_RUNTIME_SETTINGS_FILE=runtime-settings.json`
- `PSYPIC_AUDIT_LOG_STORE=file|database`
- `PSYPIC_AUDIT_LOG_FILE=audit-logs.json`
- `PSYPIC_WORKBENCH_PROJECTS_STORE=indexeddb|database`
- `PSYPIC_BOARD_DOCUMENTS_STORE=indexeddb|database`
- `PSYPIC_SEARCH_INDEX_ENABLED`

运行时配置：

- 默认 `PSYPIC_RUNTIME_SETTINGS_STORE=file`，管理端保存配置后会写入 `.data/{PSYPIC_RUNTIME_SETTINGS_FILE}`。
- 单机部署使用 file store 时，需要把项目 `.data/` 目录挂到持久化磁盘。
- 多实例生产环境设置 `PSYPIC_RUNTIME_SETTINGS_STORE=database`，通过 Prisma `runtime_settings` 模型读写，避免实例间配置漂移。
- 审计日志同理：单机可用 `PSYPIC_AUDIT_LOG_STORE=file`，生产多实例应使用 `PSYPIC_AUDIT_LOG_STORE=database`。

资产存储：

- 本地开发可使用 `ASSET_STORAGE_DRIVER=local`。
- 生产使用 `ASSET_STORAGE_DRIVER=r2|s3|minio`，并配置 `ASSET_STORAGE_ENDPOINT`、`ASSET_STORAGE_REGION`、`ASSET_STORAGE_BUCKET`、`ASSET_STORAGE_PREFIX`、`ASSET_STORAGE_ACCESS_KEY_ID`、`ASSET_STORAGE_SECRET_ACCESS_KEY`。
- R2/S3/MinIO 使用 S3-compatible client，并为临时资产、长期素材和社区公开图使用不同 prefix。
- 生产 bucket 需要最小权限：写入、读取、删除临时对象，不授予全账户管理权限。

工作台存储：

- Project / Session / VersionNode local-first 时，不要求服务端迁移。
- BoardDocument local-first 时，不要求服务端迁移；服务端化后，BoardExportAsset 必须和 ImageAsset / VersionNode 保持可回滚关联。
- 服务端化后，版本图、任务 runtime 和搜索索引要和 image_tasks / image_assets 保持可回滚关联。
- 搜索索引可重建，不作为唯一事实来源。

## 发布前检查

```bash
git diff --check
pnpm lint
pnpm typecheck
pnpm test
DATABASE_URL="postgresql://psypic:psypic@localhost:5432/psypic" pnpm prisma validate
pnpm build
pnpm e2e
APP_URL="http://127.0.0.1:3000" pnpm load:smoke
```

发布前还需要访问 `/api/health`，确认 `ok=true`，且响应中不包含 `DATABASE_URL`、`REDIS_URL` 或任何 secret。

安全检查：

```powershell
Get-ChildItem -Path app,components,lib,server,prisma -Recurse -File |
  Where-Object { $_.FullName -notmatch '\\node_modules\\|\\.next\\|\\.git\\' } |
  Select-String -Pattern 'api_key|Authorization|secret-token|localStorage|sessionStorage|indexedDB'
```

命中项必须只包含 schema、服务端 Authorization 转发、脱敏函数、手动 key 表单字段和 IndexedDB 本地历史/Prompt 收藏，不得出现真实 Key。

## 数据库

当前 Prisma schema 已覆盖：

- 用户、session、key binding、审计日志。
- 图片任务、素材、标签、相册。
- 社区作品、举报、点赞、收藏。
- 管理端运行时配置。
- 批量任务和批量项。

生产发布前必须在 staging 环境执行 migration，并验证：

- 新增字段默认值不破坏已有数据。
- `community_works` 下架、恢复、精选可回滚。
- `runtime_settings` 为空时仍能从环境变量读取默认值。
- `image_batch_items` 与 `image_tasks` 关联完整。
- `version_nodes` 与 `image_tasks`、`image_assets` 的关联可回溯。
- `board_documents` 与 `board_exports` 新增失败时，可关闭 `PSYPIC_BOARD_DOCUMENTS_STORE=database` 回退到 IndexedDB，不影响已有生成链路。
- 搜索索引 migration 失败时可关闭 `PSYPIC_SEARCH_INDEX_ENABLED` 并回退到局部筛选。

### Backend Max Usable Cut 1 迁移基线记录（2026-05-09）

目标：验证已存在的 Workbench migration
`20260509015000_workbench_models` 和 Auth migration
`20260509023000_auth_baseline` 可以在真实 PostgreSQL 路径上 apply，并确认旧
`image_tasks` 行在新增 nullable 工作台字段后仍可读取。

本机默认 `.env.example` 指向：

```powershell
DATABASE_URL="postgresql://psypic:psypic@localhost:5432/psypic"
```

2026-05-09 本地检查发现 `localhost:5432` 实际由 `turnitin-postgres` 容器提供，
容器环境为 `POSTGRES_DB=turnitin_api`、`POSTGRES_USER=turnitin`，且不存在
`psypic` role。该库不是 PsyPic 专用库，因此未对它执行 `migrate deploy`、
`migrate reset` 或任何 schema 写入。

为验证迁移 SQL，本次使用隔离的一次性 PostgreSQL 17 容器：

```powershell
docker run --name psypic-migration-cut1-check `
  -e POSTGRES_USER=psypic `
  -e POSTGRES_PASSWORD=psypic `
  -e POSTGRES_DB=psypic `
  -p 127.0.0.1:55432:5432 `
  -d postgres:17-bookworm

$env:DATABASE_URL="postgresql://psypic:psypic@localhost:55432/psypic"
pnpm prisma validate
```

验证结果：

- `pnpm prisma validate`：通过，schema 有效。
- 使用 `bfd6b3e`（Workbench/Auth 迁移前）的 schema 建立旧基线库，并插入旧任务
  `task_cut1_old`，该任务不包含 `project_id`、`session_id`、`version_node_id`。
- 旧基线库如果没有 `_prisma_migrations` 表，`pnpm prisma migrate deploy` 会返回
  `P3005`：database schema is not empty。已有业务表但未纳入 Prisma migration
  history 的 staging/生产库，必须先做一次 Prisma baseline，不能直接 deploy。
- 在隔离库中模拟已完成的 Prisma baseline history 后，`pnpm prisma migrate deploy`
  成功 apply：
  - `20260509015000_workbench_models`
  - `20260509023000_auth_baseline`
- `pnpm prisma migrate status`：`Database schema is up to date!`
- 迁移后确认表存在：
  - `workbench_projects`
  - `creative_sessions`
  - `version_nodes`
  - `image_tasks`
  - `users`
- 迁移后旧任务兼容查询：

```sql
select
  id,
  project_id is null as project_id_nullable,
  session_id is null as session_id_nullable,
  version_node_id is null as version_node_id_nullable
from image_tasks
where id = 'task_cut1_old';
```

结果：`task_cut1_old|t|t|t`。旧 `image_tasks` 行的三列均保持 `NULL`，不需要回填假
project/session/version node。

迁移失败或 Workbench 服务端化未就绪时，运维回退开关仍可用：

```powershell
$env:PSYPIC_WORKBENCH_PROJECTS_STORE="indexeddb"
```

`PSYPIC_WORKBENCH_PROJECTS_STORE=indexeddb|database` 已记录在本文环境变量清单，
`.env.example` 默认值为 `database`。Cut 3 的 Workbench API 路由实现后，
`indexeddb` 模式必须返回 `503` + `Retry-After`，前端 fallback 到 IndexedDB 本地工作台。

## 健康检查

上线后检查：

- `/api/session` 不返回 API Key。
- `/api/health` 返回 `db`、`redis`、`temp_asset`、`storage`、`runtime_settings` 检查结果。
- `/api/images/generations` 可使用测试 Key 完成 1 张小图。
- `/api/images/edits` 可上传 1-2 张参考图。
- `/api/community/works` 只返回公开且未下架作品。
- `/admin` 仅管理员可访问。
- 审计日志包含生成请求、上游 request id 和管理员操作。

## 回滚步骤

1. 记录当前版本 tag、镜像 tag 或部署产物 id。
2. 发布前把上一版标记为 `previous-stable`。
3. 若健康检查失败，立即切回上一版产物。
4. 回滚后执行 `/api/session`、生成接口和社区列表健康检查。
5. 如果失败原因来自数据库 migration，只允许执行事先验证过的回滚脚本。

工作台相关回滚：

- Job queue 切换失败时，先暂停 worker，再让 BFF 回到 in-memory 或只读任务查询模式。
- Version graph migration 失败时，保留旧任务和素材数据，不执行级联删除。
- BoardDocument / BoardExport migration 失败时，保留已生成 ImageAsset，不删除用户画布导出的 reference/mask 文件，只回滚 metadata 指针。
- 搜索索引异常时关闭搜索索引开关，保留精确 asset/task/project 查询。
- 对象存储 prefix 迁移失败时，不移动旧对象，只回滚 metadata 指针。

## 压测建议

最小压测范围：

- 图片生成 BFF 并发限制。
- 批量任务创建和队列状态查询。
- 素材库分页和标签过滤。
- 社区信息流 latest/popular/featured 排序。
- 管理端举报列表。

压测必须使用测试 Key 和测试 bucket，不使用生产用户 Key。

仓库内置最小 smoke/load 脚本：

```bash
APP_URL="https://staging.example.com" PSYPIC_LOAD_REQUESTS=100 PSYPIC_LOAD_CONCURRENCY=8 pnpm load:smoke
```

默认脚本只压 `/api/health` 和社区公开信息流，不会触发真实生图。需要压生成链路时必须显式提供测试 session cookie，并开启生成：

```bash
APP_URL="https://staging.example.com" \
PSYPIC_LOAD_COOKIE="psypic_session=..." \
PSYPIC_LOAD_RUN_GENERATION=true \
PSYPIC_LOAD_REQUESTS=10 \
PSYPIC_LOAD_CONCURRENCY=2 \
pnpm load:smoke
```

生成链路压测必须在 staging 使用测试 Key 和测试 bucket，压测前确认 Sub2API 额度和成本上限。
