# PsyPic 整站 UI 系统设计

## 目标

为 PsyPic 建立一套统一、专业、可持续演进的整站 UI 系统。第一眼必须像成熟商业创作产品，第二眼能看出它不是营销落地页，也不是通用后台，而是围绕商业生图、灵感复用和版本创作构建的专业工作环境。

这份设计文档覆盖：

- 全局产品壳与导航。
- 工作台主界面。
- 灵感社区与作品详情。
- 素材详情。
- 设置页。
- 管理台。
- 移动端适配。
- 通用视觉、组件、状态和交互规则。

## 当前产品事实

- 默认首页已经是新版 `chat studio`，旧版画廊三栏只作为 fallback，见 `components/creator/CreatorWorkspace.tsx`。
- 工作台已经拆成三层语义：
  - 左侧：项目、对话、分支等上下文导航。
  - 中间：对话式版本流、结果、Composer。
  - 右侧：参数、模板、素材、历史和 Inspector。
- 站点级独立页面已存在：
  - `/` 工作台
  - `/community` 灵感社区
  - `/community/works/[workId]` 作品详情
  - `/library/[assetId]` 素材详情
  - `/settings` 设置
  - `/admin` 管理台

因此，新的 UI 系统不能再把所有入口都挤进一个“漂亮顶栏”里，而要以清晰层级组织信息。

## 设计原则

- 首屏即工作台，不引入营销首页。
- 高级感来自比例、留白、材质和层级，而不是渐变噪音和装饰图形。
- 全局导航只承担跨页面跳转，不承载工作台内部工具。
- 工作台内部继续使用上下文导航和 Inspector，不与站点导航混层。
- 社区不是独立娱乐流，必须服务创作回流。
- 所有页面共用同一套按钮、表单、卡片、状态和字体规则。
- 移动端不是桌面端截图缩放，必须重新组织层级。

## 信息架构

### 四层结构

PsyPic 的 UI 分为四层：

1. 全局层  
   回答“我在 PsyPic 的哪个系统页面”。

2. Studio 上下文层  
   回答“我在当前工作台的哪个项目、对话或分支”。

3. 任务层  
   回答“我现在正在做什么创作动作”。

4. 工具层  
   回答“我可以调哪些参数、看哪些素材、复用哪些历史”。

### 各层职责

```text
全局层
  工作台 / 灵感社区 / 设置 / 管理台(按权限)

Studio 上下文层
  项目 / 对话 / 分支 / 当前上下文

任务层
  当前对话 / 版本流 / 结果 / Composer / 主 CTA

工具层
  参数 / 模板 / 素材 / 历史 / 版本图 / 批量
```

### 禁止混层

以下内容不进入全局导航：

- 项目
- 对话
- 分支
- 参数
- 商业模板
- 历史与素材
- 批量工作流
- 版本流

这些都属于工作台内部能力，而不是站点一级页面。

## 全局产品壳

### 顶部导航

全局壳使用轻量、悬浮、粘性的顶部导航条。

结构：

```text
左：品牌区
中：主导航
右：工具区
```

### 品牌区

- 显示 `PsyPic`。
- 配短副标题，如 `商业创作台` 或 `商业创作系统`。
- 品牌区负责识别与气质，不承担功能密度。

### 主导航

主导航仅放两个一级入口：

- `工作台`
- `灵感社区`

原因：

- 它们是产品最核心的两个系统级场景。
- 一个负责创作，一个负责灵感回流。
- 只有这两个页面真正需要被长期固定在主导航中心。

### 工具区

右侧工具区放：

- `设置`
- `管理台`，仅管理员可见
- 未来预留 `账号 / session / 环境` 状态位

原因：

- 设置是 utility，不是主任务。
- 管理台是权限页，不属于普通用户主流程。
- 右侧工具区更符合成熟 SaaS 产品的导航语义。

### 当前页高亮

- 当前页高亮使用低对比 pill 或细描边胶囊，不用饱和底色 tab。
- 仅主导航和工具页入口需要高亮。
- Studio 内部项目和对话继续使用自己的 active 规则，不复用全局导航视觉。

### 与现有工作台的关系

- 工作台左下角目前存在 `社区 / 设置` 临时入口。
- 新全局壳上线后，这类重复入口应从 Studio 侧栏中移除或降级。
- 页面内部 `返回创作台` 按钮应逐步收敛，不再依赖孤立页头来表达导航。

## 视觉系统

### 气质关键词

- 专业
- 克制
- 干净
- 精密
- 可信
- 以图片和创作为中心

### 背景与材质

- 主背景使用中性浅灰、冷白或低噪声雾面底色。
- 导航和浮层使用半透明表面、细描边、极轻阴影。
- 避免大面积营销式渐变、光斑、球体、霓虹描边。

### 色彩

- 主色只用于焦点、CTA、活动态，不统治整个界面。
- 工作台主背景应让图片成为视觉中心。
- 社区可以略微更视觉化，但不能牺牲内容可读性。
- 管理台颜色偏稳重，危险动作需要清晰风险提示。

### 字体

- 中文使用系统无衬线。
- 英文、数字、参数摘要同样使用系统无衬线。
- 不引入强装饰性字体。
- 标题、标签、参数元信息的层级必须通过字重、字号和间距区分，而不是通过大量颜色区分。

### 圆角、阴影、描边

- 全站卡片圆角保持克制，优先使用 8px 级别。
- 阴影轻，不制造漂浮感噪声。
- 通过描边与表面反差建立精度感。

## 通用组件规则

### 按钮系统

统一三层：

- `Primary`
  - 用于当前区域唯一主动作，如 `生成图片`、`保存运行时配置`、`生成同款草稿`
- `Secondary`
  - 用于次级操作，如 `恢复参数`、`继续编辑`、`下载`
- `Ghost / Utility`
  - 用于顶部工具、轻操作、收纳式入口

规则：

- 同一块区域内只能有一个视觉主按钮。
- 文本按钮用于明确动作，图标按钮用于高频工具。
- 危险操作必须有独立样式语义。

### 表单控件

- 模式和质量优先使用 segmented control。
- 选择型参数优先使用 select，不做大量自定义轮子。
- 高级参数默认折叠。
- 表单中的安全说明使用低干扰说明文字，不与错误态混用。

### 卡片

- 卡片只用于结果、作品、模板、指标、浮层信息块。
- 不把整页 section 全部套成厚重卡片墙。
- 图片卡片必须固定比例，避免布局跳动。

### 标签和状态

- `template-pill`、状态条、标签 chips 全站统一尺寸和节奏。
- 标签应服务筛选、分类、状态识别，不做纯装饰。
- 成功、加载、失败、运行中、精选、可同款等状态都应有统一语义层级。

## 工作台设计

### 标准形态

工作台以新版 `chat studio` 为标准形态，legacy 画廊三栏不是主设计基准。

结构：

```text
Global Shell
└─ Workbench
   ├─ Studio Sidebar
   │  ├─ 项目
   │  └─ 对话 / 分支
   ├─ Chat Workspace
   │  ├─ 当前对话头部
   │  ├─ 对话式版本流
   │  ├─ 任务状态 / partial preview
   │  └─ 固定 Composer
   └─ Inspector
      ├─ 参数
      ├─ 参考图 / 遮罩
      ├─ 商业模板
      ├─ 版本流 / 分支图
      ├─ 批量工作流
      └─ 素材与历史
```

### 左侧 Studio Sidebar

职责：

- 展示项目列表。
- 展示当前项目下的对话、主线和分支。
- 只负责创作上下文切换。

规则：

- 左侧不再承载全局站点导航。
- 项目和对话必须区分视觉层级。
- 激活态强调“当前上下文”，而不是“当前页面”。
- 新建对话入口要清楚、可独立识别。

### 中间 Chat Workspace

职责：

- 展示当前对话和版本流。
- 成为用户主要视觉焦点。
- 保持生成动作与结果的连续性。

规则：

- `当前对话` 头部简洁，不做站点导航替代。
- 对话消息结构明确区分用户 Prompt 与 PsyPic 结果。
- 结果卡是对话的一部分，不是脱离上下文的图库。
- Composer 永远可见，且主操作按钮始终在同一位置。

### 版本流与分支

版本流是新版工作台的核心能力，必须在视觉和交互上被清楚表达。

规则：

- 每个节点显示：
  - Prompt 摘要
  - 分支标签
  - 参数摘要
  - 结果图
  - request id / tokens 等必要元信息
- `回到版本`、`恢复参数`、`从此分叉` 必须是明确动作，不混入普通结果卡操作。
- 分支图是创作结构视图，不是装饰图形。
- 用户必须能理解：
  - 当前在哪个版本
  - 新结果是否来自分叉
  - 恢复参数不会破坏现有版本

### Composer

职责：

- 提供 Prompt 输入。
- 汇总当前关键参数。
- 承载优化 Prompt、收藏 Prompt、生成图片等主流程动作。

规则：

- 主 CTA 只有一个：`生成图片`。
- 参数上下文只显示关键摘要，不复制整套 Inspector。
- 输入区必须稳定、固定、无跳动。
- 错误提示就地显示，且尽量提供下一步动作。

### Inspector

职责：

- 承载参数、模板、参考图、历史素材、批量、版本结构等工具性内容。

规则：

- Inspector 是工具层，不与左侧对话导航抢语义。
- 模板和参数永远服务当前对话，不单独成为站点主页面。
- 素材与历史强调“复用”，不是只做存档。
- 批量工作流在信息架构中属于增强功能，不盖过主创作流。

### Legacy 画廊工作台

- legacy 版本保留功能兜底作用。
- 视觉系统需与新版一致。
- 不作为未来主要 UI 设计方向。

## 灵感社区设计

### 定位

灵感社区是创作上游，不是泛娱乐瀑布流。

### 首页结构

- 顶部使用统一全局壳。
- 页面主体包含：
  - 页面标题区
  - 排序筛选
  - 场景筛选
  - 标签筛选
  - 作品卡片网格

### 作品卡

每张作品卡至少包含：

- 封面图
- 标题
- 场景 / 精选标记
- 标签
- 查看作品
- 点赞
- 收藏
- 可同款状态

规则：

- 图片是主视觉，但卡片信息必须服务复用。
- hover 或焦点态应突出详情和复用动作，而不是只强调互动数。
- 同款能力必须足够清楚，不能埋在详情里。

### 空状态

- 明确说明社区内容来自创作台发布。
- 提供回到工作台或开始创作的动作。

## 作品详情设计

### 页面目标

把浏览兴趣尽快转成可执行创作动作。

### 布局

- 主图区域
- 信息与操作侧栏

### 侧栏内容

- 标题
- 精选 / 场景 / 标签
- 点赞、收藏、举报
- Prompt 公开状态
- 同款生成主动作
- 套用到创作台的次动作

规则：

- 主 CTA 是 `生成同款草稿`。
- `套用到创作台` 必须紧跟其后。
- Prompt 未公开时也要清楚解释剩余可复用内容。
- 举报入口存在，但不抢主流程。

## 素材详情设计

### 页面目标

作为“结果复用页”存在，而不是纯信息页。

### 页面结构

- 统一全局壳
- 左侧预览或主图
- 右侧素材信息

### 核心动作

- 下载
- 继续编辑

规则：

- `继续编辑` 是主要动作。
- 参数、格式、tokens、耗时都应作为专业元信息展示，但不压过图片和复用动作。
- 无权限或不存在时给出明确错误和回流路径。

## 设置页设计

### 定位

设置页属于 utility page，强调安全、清晰、可信。

### 结构

- 统一全局壳
- 页面标题和说明
- 设置表单
- 保存反馈

规则：

- 安全说明应始终靠近输入项。
- 成功与失败反馈明确，但不过度喧闹。
- 页面风格应比社区更克制，比管理台更友好。

## 管理台设计

### 定位

管理台是运营控制台，不是普通用户工作区的一部分。

### 结构

- 统一全局壳
- 页面头部
- usage 指标卡
- 运行时配置
- 举报列表
- 审计日志

### 规则

- 页面强调密度、控制感、可信赖感。
- 指标区要快速扫读。
- 危险动作如 `下架` 必须有清晰语义。
- 审计日志弱化视觉，但保留结构清晰度。
- 管理台视觉不追求社区感，而要追求稳定和可控。

## 移动端设计

### 总原则

- 保主流程，不复制桌面布局。
- 重新组织导航与工具层级。

### 全局层

- 顶部保留品牌和全局导航入口。
- 主导航折叠为抽屉或轻量菜单。

### 工作台

- 左侧项目 / 对话改为侧滑或全屏抽屉。
- Inspector 改为底部抽屉或分段面板。
- Composer 固定在底部。
- 中间结果区优先保证浏览和主动作可见。

### 社区与详情

- 社区筛选收进 chips + 抽屉。
- 作品详情优先主图和同款主 CTA。
- 素材详情优先预览和继续编辑。

### 管理台

- 移动端只要求基础可用。
- 复杂审核和配置仍以桌面效率为主。

## 状态设计

### 空状态

- 不写长篇说明。
- 必须提供可直接点击的下一步动作。

### 加载状态

- 使用稳定骨架或固定尺寸占位。
- 不改变最终布局结构。

### 错误状态

- 用可理解文案说明问题。
- 能补动作就补动作。
- 不把 request id 直接当用户主信息，但可在专业场景下展示。

### 成功状态

- 生成成功后必须突出复用动作：
  - 下载
  - 继续编辑
  - 作为参考图
  - 发布

## 动效规则

- 只做必要动效。
- hover、active、展开收起、导航高亮、任务状态更新可使用轻量过渡。
- 避免漂浮、发光、夸张缩放和过度弹性动画。
- 分支和版本流的动效优先帮助理解结构，而不是制造炫技感。

## 可访问性

- 所有主操作按钮必须有清晰文本或可读 aria-label。
- 当前页、当前对话、当前项目、当前版本必须有可辨识状态。
- 色彩不是唯一状态表达方式。
- 错误消息、任务状态、生成状态需要可被辅助技术识别。

## 页面级落地要求

### 工作台

- 上全局壳。
- 保留左中右三层语义。
- 左下角重复的社区/设置入口后续移除或降级。

### 社区

- 接入全局壳。
- 卡片、筛选和 CTA 统一视觉系统。

### 作品详情

- 接入全局壳。
- 强化同款主动作。

### 素材详情

- 接入全局壳。
- 弱化“返回页头”，强化全局导航和继续编辑动作。

### 设置

- 接入全局壳。
- 保留安全说明与 BFF 语义。

### 管理台

- 接入全局壳。
- 按权限显示入口。

## 非目标

- 不把首页改成营销官网。
- 不把所有功能做成一级导航。
- 不做大面积品牌视觉表演。
- 不改变现有 API 契约和版本节点数据模型。
- 不在本设计阶段引入模型广场、评论社交、团队协作等新产品范围。

## 验收标准

- 新用户进入站点后，3 秒内能理解站点级导航与工作台内部导航的区别。
- 用户进入工作台后，不会把项目/对话误认成全局页面导航。
- 社区、详情、设置、管理台具备统一产品壳，不再像孤立页面。
- 全局导航只保留真正的系统级页面入口，没有参数、模板、素材等伪一级项。
- 新版 chat studio 成为设计标准形态，legacy 画廊作为 fallback 继承同一视觉系统。
- 工作台、社区、详情、设置、管理台在按钮层级、表单风格、卡片节奏和状态样式上统一。
- 整体视觉效果像成熟商业创作软件，而不是营销落地页或通用后台。
