# 17. 测试与验收用例

## 目标

本文件用于开发阶段验证 PsyPic 是否真的可用，而不是只完成 UI 或接口草稿。

覆盖：

- Sub2API 一键导入。
- session 和密钥安全。
- 文生图。
- 图生图。
- TempAsset 下载。
- IndexedDB 基础历史。
- 参数校验。
- 错误处理。
- 商业尺寸预设。
- MVP/V1/封板验收。
- Project Workspace、Version Graph、Job Runtime 和 Search / Command Surface。

## 测试类型

| 类型 | 工具 | 阶段 |
| --- | --- | --- |
| 单元测试 | Vitest | 每个阶段 |
| API 测试 | Vitest / Supertest / fetch | `v0.2+` |
| 前端组件测试 | Testing Library | `v0.1+` |
| E2E | Playwright | `v0.3+` |
| 手工验收 | 浏览器 | 每个阶段 |
| 安全检查 | 脚本 + 人工检查 | `v0.2+` |

## 基础命令

```bash
pnpm lint
pnpm typecheck
pnpm test
pnpm build
```

E2E：

```bash
pnpm e2e
```

首次运行 E2E 前安装 Chromium：

```bash
pnpm e2e:install
```

当前 `pnpm e2e` 使用受保护的本地 E2E session 入口和 fake Sub2API，上线前仍需用真实 Sub2API、对象存储和 staging 数据库补一次人工验收。

## API 测试用例

### Health

请求：

```http
GET /api/health
```

期望：

- HTTP 200。
- `data.ok=true`。
- 返回 db、redis、temp_asset 状态。
- `storage` 只在配置 R2/S3/MinIO 时返回；未配置时应为 `skipped` 或不返回，不能阻塞 MVP。

### Import Exchange 成功

请求：

```http
POST /api/import/exchange
Content-Type: application/json

{
  "import_code": "valid_one_time_code"
}
```

期望：

- HTTP 200。
- 返回 `session_bound=true`。
- 设置 `psypic_session` HttpOnly cookie。
- 响应体不包含 `api_key`。
- 响应头包含 `Cache-Control: no-store`。

### Import Exchange 重复消费

步骤：

1. 使用同一个 `import_code` 请求成功一次。
2. 再次使用同一个 `import_code` 请求。

期望：

- 第二次 HTTP 400 或 403。
- 错误码为 `invalid_import_code` 或 `forbidden`。
- 不创建新的 session。

### Session

请求：

```http
GET /api/session
Cookie: psypic_session=...
```

期望：

- HTTP 200。
- 返回 `authenticated=true`。
- 返回 binding、limits、features。
- 不返回 API Key。

### 文生图成功

请求：

```http
POST /api/images/generations
Content-Type: application/json

{
  "prompt": "为一瓶透明玻璃香水生成高端电商主图...",
  "model": "gpt-image-2",
  "size": "1024x1024",
  "quality": "medium",
  "n": 1,
  "output_format": "png",
  "background": "auto",
  "moderation": "auto"
}
```

期望：

- HTTP 200。
- 返回 `task_id`。
- 返回至少 1 个 `images[]`。
- 每个 image 有 `asset_id` 和 `/api/assets/{asset_id}` URL。
- 返回 `request_id`。
- 如果 Sub2API 返回 upstream request id，响应中透出。
- 记录 usage 或 estimated cost。

### 文生图参数错误

场景：

- prompt 为空。
- `n` 超过 limits。
- `background=transparent`。
- 非法 size。

期望：

- HTTP 400。
- 错误码 `invalid_parameter`。
- `details.field` 指向错误字段。
- 不调用 Sub2API。

### 图生图成功

请求：

```http
POST /api/images/edits
Content-Type: multipart/form-data

prompt=保持产品主体完全一致...
image=<product.png>
size=1024x1024
quality=medium
n=1
```

期望：

- HTTP 200。
- 返回 `task_id` 和图片。
- 上传文件被校验。
- 响应可下载。

### 遮罩编辑成功

`v0.6` 起测试。

请求：

```http
POST /api/images/edits
Content-Type: multipart/form-data

prompt=只替换涂抹区域...
image=<product.png>
mask=<mask.png>
size=1024x1024
quality=medium
n=1
```

期望：

- HTTP 200。
- `mask` 是 PNG 文件。
- BFF 转发给 Sub2API 的 multipart 同时包含 `image` 和 `mask`。
- 响应不泄露 API Key。

### 图生图上传错误

场景：

- 未上传 image。
- 上传非图片。
- 上传超大文件。
- 上传非 PNG mask。
- 上传尺寸与参考图不一致的 mask。

期望：

- 未上传 image：HTTP 400。
- 非图片：HTTP 415。
- 超大文件：HTTP 413。
- 非 PNG mask：HTTP 415，`details.field=mask`。
- mask 尺寸不一致：HTTP 400，`details.field=mask`。
- 错误提示可读。

### 任务状态查询

`v0.5` 起测试。

请求：

```http
GET /api/tasks/{task_id}
Cookie: psypic_session=...
```

期望：

- 本人访问 HTTP 200。
- 返回 `id`、`type`、`status`、`prompt`、`params`、`created_at`、`updated_at`。
- 成功任务返回 images、usage、duration_ms。
- 失败任务返回脱敏后的 `error.code` 和 `error.message`。
- 非本人或不存在任务返回 404。
- 未登录返回 401。

### 任务取消

`v0.5` 起测试。

请求：

```http
POST /api/tasks/{task_id}
Cookie: psypic_session=...
```

期望：

- queued/running 任务返回 HTTP 200 且 `status=canceled`。
- succeeded/failed/canceled 任务返回当前状态，不删除结果。
- 只能取消当前 session 所属用户的任务。
- 前端任务状态区显示取消后的状态，并提供重新生成入口。

### 工作台项目与版本图

`v1.1+` 服务端化后测试；local-first 阶段用组件 / hook 测试覆盖同样语义。

请求：

```http
GET /api/workbench/projects
GET /api/workbench/version-graph?project_id=proj_xxx&session_id=session_xxx
```

期望：

- 只返回当前用户可访问项目。
- 项目包含 active session、排序和折叠状态。
- 版本节点包含 parent、prompt snapshot、params snapshot、task id、asset ids。
- 从旧节点 fork 后，旧后续节点仍存在。
- 非本人项目或版本节点返回 404。

### Board Mode Board API

`v1.1+` 服务端化后测试；local-first 阶段用 BoardDocument 序列化、组件和 E2E 覆盖同样语义。

步骤：

1. 创建 `BoardDocument`，包含一个 image layer、一个 stroke layer 和一个 mask layer。
2. 更新图层排序、隐藏、锁定和 transform。
3. 导出 `reference_png`。
4. 导出 `mask_png`。
5. 用导出的 reference / mask 提交 `/api/images/edits`。

期望：

- Board 只能访问当前用户有权限的 project、session 和 source asset。
- 导出返回 `board_export_id`、`asset_id`、width、height、pixel_ratio。
- `reference_png` 可作为 edits 的 `image`。
- `mask_png` 可作为 edits 的 `mask`，且 MIME 为 PNG。
- 新 version node 包含 `board_document_id`、`board_snapshot`、`board_export_asset_id`。
- 超大画布、超多图层或超大 stroke JSON 返回 400，不调用上游。

### 图片任务并发限制

`v0.5` 起测试。

场景：

- 同一用户已有 `queued` 或 `running` 图片任务。
- 再次请求文生图、图生图或流式文生图。

期望：

- HTTP 429。
- 错误码 `rate_limited`。
- 错误提示说明当前已有任务正在运行。
- 不调用 Sub2API。
- 默认限制为每用户 1 个 active 图片任务，可通过 `PSYPIC_MAX_ACTIVE_IMAGE_TASKS_PER_USER` 调整。

### 服务端历史

`v0.7` 起测试。

请求：

```http
GET /api/history?limit=30&cursor=task_xxx
Cookie: psypic_session=...
```

期望：

- 未登录返回 HTTP 401，错误码 `unauthorized`。
- 只返回当前用户已成功并且有图片的任务。
- 历史项包含 `task_id`、`type`、`prompt`、`params`、`thumbnail_url`、`images`、`usage`、`created_at`。
- 如果任务有 upstream request id，历史项透出 `upstream_request_id`。
- `next_cursor` 使用当前页最后一条 `task_id`，下一页不跳过记录。
- 响应不包含 API Key、session cookie 或 key binding secret。

### 素材库

`v0.7` 起测试。

请求：

```http
GET /api/library?limit=30&favorite=true&tag=电商主图&q=香水
Cookie: psypic_session=...
```

期望：

- 未登录返回 HTTP 401，错误码 `unauthorized`。
- 只返回当前用户已成功任务中的图片资产。
- 素材项包含 `asset_id`、`task_id`、`prompt`、`url`、`thumbnail_url`、`favorite`、`tags`、`created_at`。
- `favorite=true` 只返回收藏资产。
- `tag` 能按规范化标签过滤。
- `q` 能匹配 prompt、task id、asset id 和标签。
- 响应不包含 API Key、session cookie 或 key binding secret。

请求：

```http
PATCH /api/library/{asset_id}
Content-Type: application/json
Cookie: psypic_session=...

{
  "favorite": true,
  "tags": ["电商主图", "香水"]
}
```

期望：

- 本人资产返回 HTTP 200，并返回更新后的素材项。
- tags 去重、去空白。
- 非本人或不存在资产返回 404。
- `favorite` 非布尔值或 `tags` 非字符串数组时返回 HTTP 400。

## 前端 E2E 用例

### 首屏创作台

步骤：

1. 打开 `/`。
2. 查看首屏。

期望：

- 默认进入创作台。
- 左侧参数栏可见。
- 中间结果区可见。
- 右侧历史/详情区可见。
- 商业模板入口可见。
- 没有营销 hero 抢占首屏。

### 画廊工作台与版本流

步骤：

1. 在创作台输入 Prompt 并生成。
2. 修改 Prompt 或尺寸后再次生成。
3. 在版本流点击第一条节点的恢复参数。
4. 点击第一条节点的从此分叉，再输入新 Prompt 生成。

期望：

- 中间主区以结果画廊为视觉核心，能容纳多版本图片。
- 每次生成后版本流新增节点，节点展示 prompt 摘要和参数摘要。
- 恢复参数只恢复 Composer 和参数控件，不删除任何历史节点。
- 从旧节点分叉生成时，旧后续节点保留，新节点显示为新的分支。
- 分支图能展示主线和分支关系。
- Inspector 能展示当前节点的 request id、耗时和参数快照。

### Board Mode 模式

步骤：

1. 在工作台切到 Board Mode。
2. 从素材库或结果卡把两张图片加入画布。
3. 拖动、缩放、旋转其中一张图片。
4. 添加手绘标注和一段文字 brief。
5. 启用 mask layer 涂抹局部区域。
6. 点击“作为参考图编辑”。
7. 补充 prompt 后提交生成。

期望：

- 画布非空，图片层、文字层、手绘层和 mask 层都能在图层列表中选择。
- Inspector 能展示当前图层的 transform、brush 或 export 控制。
- 导出的 reference PNG 非空，尺寸符合 BoardDocument。
- mask PNG 只包含 mask layer，不混入普通标注。
- Composer 切到图生图模式并带入 board export。
- 生成成功后版本流新增节点，节点能恢复 board snapshot。
- 移动端至少支持查看画布、添加图片、移动缩放、画笔标注和导出。

### Project Sidebar 与 Task Dock

步骤：

1. 创建或展开多个项目。
2. 在非 active 项目 header 上展开和折叠。
3. 在某个项目会话中触发生成。
4. 打开 Task Dock 查看任务状态。
5. 取消 running/queued 任务后重试。

期望：

- 项目位置不因 active 切换跳动。
- 多项目展开状态互不影响。
- Task Dock 展示 queued / running / partial / failed / canceled。
- 取消和重试不会暴露 API Key，也不会覆盖旧版本节点。

### 搜索与命令面板

步骤：

1. 唤起搜索 / 命令面板。
2. 搜索 prompt 关键词、board title、asset id、task id、request id。
3. 执行“新建 Board Mode”“从当前节点分叉”“导入素材”“打开设置”等命令。

期望：

- 搜索结果按 project / session / version / board / asset / task 分组。
- 命中项能回到对应工作台上下文。
- 命令执行后不丢当前 Composer 和 Inspector 状态。

### 手动配置并生成

步骤：

1. 打开设置。
2. 输入 Sub2API Base URL 和 API Key。
3. 回到创作台。
4. 输入测试 prompt。
5. 点击生成。

期望：

- 按钮进入 loading。
- 生成成功后展示结果卡。
- 可下载图片。
- 历史区出现记录。

### 服务端素材库同步

步骤：

1. 完成至少一次生成。
2. 在右侧历史与素材面板点击同步素材库。
3. 查看素材卡片并点击收藏。

期望：

- 前端请求 `/api/library?limit=30`。
- 素材卡片展示 asset id、prompt、task id、tags 和 usage/耗时。
- 点击收藏调用 `PATCH /api/library/{asset_id}`。
- 收藏状态更新后按钮变为取消收藏。
- 素材库不可用时仍保留本地历史继续编辑。

### 图片详情页

`v0.7` 起测试。

步骤：

1. 在创作台同步素材库。
2. 点击素材卡片的详情。
3. 在详情页点击下载或继续编辑。

### 自动化测试

### 真机走查 4 项

| 序 | 场景 | 预期 |
| --- | --- | --- |
| 1 | 在某项目下生成一条 session → 右键该 session → 分叉到同一工作树 | active 切到该 branch，Composer 自动填回 prompt + params；ChatHeader / VersionStream 显示「分叉中」pill；toast「已分叉，请在 Composer 中输入新 prompt」 |
| 2 | 同上但点「派生到新工作树」 | sidebar 列表新增 `${源项目名} · 派生` 项目（IndexedDB 持久化）；active 自动切到新项目；activeConversationId="new"；Composer 已填 prompt+params；toast「已派生到新项目，可在 Composer 中继续」 |

### 关键不变量

- 派生不复制节点 / 不改原项目数据；新项目从 0 个节点开始（用户点生成时才长第一个）
- 分叉**留在同项目**；派生**起新项目** —— 两者完全互不交叉
- 没有 latestNode 的 session（branchSummaries 里 latestNode 为 null）：fork/derive 都会 setErrorMessage 兜底，不切状态
- IndexedDB 不可用时 derive 走 fallback：只填 Composer 不切项目，toast 仍 show（提示用户已经派生了，但因为环境限制没真正落库）

### 真机走查 7 项（4 实做 + 3 占位）

| 序 | 场景 | 预期 |
| --- | --- | --- |
| 1 | 右键 session → 置顶对话 | 该 session 立即跳到「置顶」section（在所有时间桶之前）；toast「已置顶对话」；菜单文案变成「取消置顶」 |
| 2 | 同上点「取消置顶」 | session 回到原时间桶位置；toast「已取消置顶」 |
| 3 | 右键 session → 重命名对话 | 弹 SessionRenameDialog，输入框预填 prompt 摘要；输入新标题保存 → row 标题立刻变；toast「已重命名」。再次重命名留空保存 → 标题恢复 prompt 摘要；toast「已恢复默认标题」 |
| 4 | 右键 session → 归档对话 | session 立即从列表消失；底部出现「显示归档（1）」toggle；toast「已归档对话」。点 toggle 显示归档 → 该 session 重新可见；点其菜单的「恢复对话」→ 回到时间桶；toast「已恢复对话」 |
| 5 | 在 session 有新节点未点开时 | row 左侧出现蓝点 indicator + 标题加粗。点 row 切到该 branch → indicator 消失（自动 markRead）。再点菜单「标记为未读」→ indicator 重新出现；toast「已标记为未读」 |
| 6 | 右键 session → 在资源管理器中打开 | toast「在资源管理器中打开」为桌面端独占功能 · 请使用 Tauri 客户端 |
| 7 | 右键 session → 复制工作目录 / 在迷你窗口中打开 | 同上文案模板 |

### 关键不变量

- branch-meta `branchId` 用 version-graph 全局唯一 ID 做 keyPath：换项目 / 换刷新 都能跨设备认得（同一 IndexedDB origin 内）
- `customLabel = ""` normalize 为 undefined（恢复默认）；不存空字符串避免 `customLabel ?? prompt` 短路失败
- `hasUnread` 是派生不是存的：`!lastReadAt || lastReadAt < latestNode.createdAt`；意味着「之前读过但又有新节点」会自动重新未读
- 归档不删数据：isArchived=true 仅 sidebar filter，version-graph / 节点全部不动
- 置顶 + 归档 + 未读三态正交，可同时为 true（已归档但置顶 + 未读：从「显示归档」toggle 进入后能看到所有效果叠加）
- IndexedDB 不可用时所有 5 个 callback 都 fallback 到 in-memory only：当前 session 内体验完整，刷新丢失
