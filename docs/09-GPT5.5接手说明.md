## 固定边界

- PsyPic 是独立生图站，不嵌进 Sub2API 管理后台。
- 产品必须同时做商业创作台和灵感社区；优先赢商业创作效率，社区作为作品沉淀、同款生成和增长入口。
- V1.0 后的产品语言统一为 project / session / version node / asset / job，不再只按“单次生图页面”理解创作台。
- 除非用户明确指定其他栈，默认使用 Next.js App Router + TypeScript + BFF Route Handlers。
- MVP 使用 Images API：`/v1/images/generations` 和 `/v1/images/edits`。
- 以下 `v0.x` / `v1.0` 条目是历史版本边界，不代表当前仍停在 `v0.4`。
- `v0.3` 是文生图 alpha，不是公开试用版本。
- `v0.4` 是公开试用 MVP，包含图生图、单参考图、继续编辑和基础历史。
- `v0.5` 才开始做流式 partial preview 和任务队列。
- `v0.8` 开始强化商业模板和 Prompt 助手。
- `v0.9` 建立社区最小发布模型、基础作品详情、基础同款生成和审核入口。
- `v1.0` 必须包含完整灵感社区信息流、作品详情增强和一键同款生成增强。
- 前端不直接持有长期 Sub2API API Key。
- Sub2API 一键导入必须采用短期 ticket / `import_code` + PsyPic BFF 后端代持。
- 导入交换响应不得返回 `api_key`；后续前端只带 `psypic_session` cookie 调 PsyPic API。
- MVP 历史和图片缓存可先放 IndexedDB，但 BFF 仍必须负责 session、密钥代持、参数校验、限流和转发。

## 新会话第一阶段任务

先确认当前基线：

1. `v0.1-v1.0` 主要产品切片已完成：文生图、图生图、遮罩、任务、历史、素材库、模板、社区、批量、管理端和移动端 smoke。
2. 当前文档已加入 [19-图片工作台改造蓝图.md](./19-图片工作台改造蓝图.md)，后续统一按 project / session / version node / asset / job 讨论；Board Mode 专项见 [20-拼图画布模式改造方案.md](./20-拼图画布模式改造方案.md)。
3. 下一阶段通常二选一：生产环境演练，或 `v1.1+` 图片工作台平台化。

`v1.1+` 第一批可选开发任务：

1. 先决定 Project / Session / VersionNode 是否继续 local-first，还是进入服务端化。
2. 为 WorkbenchProject、CreativeSession、VersionNode 补模型 / API / 测试计划。
3. 把 Task Dock 与 `/api/tasks` 统一成 job runtime 视图。
4. 设计 Search / Command Surface，先做本地搜索还是直接服务端索引。
5. 做 Board Mode 本地原型：多素材拼图、手绘、mask、reference PNG 导出和 version node snapshot。
6. 抽出 provider boundary，确保 Sub2API 字段不污染工作台快照。
7. 回归 [07-验收清单.md](./07-验收清单.md) 的“图片工作台化验收”和 [17-测试与验收用例.md](./17-测试与验收用例.md) 的新增回归。

## 需要先问用户的事项

- 是否接受默认 Next.js 技术栈，或明确改为 Nuxt、Vue SPA、React SPA 等其他方案。
- ORM 使用 Prisma 还是 Drizzle。
- 对象存储使用 R2、S3 还是 MinIO。
- PsyPic 是否需要独立用户系统，还是完全依赖 Sub2API 导入 session。
- MVP 是否必须部署上线，目标平台是 Vercel、Cloudflare、Render、自有服务器还是其他。
- 手动 API Key 模式是否只用于开发，还是也面向正式用户开放。
- 公开试用阶段是否允许 4K、`moderation=low`、以及单次 `n > 1`。
- 社区发布是否默认审核，是否允许用户隐藏 prompt、参数和参考图。
- 当前优先做生产演练，还是先做 `v1.1+` 图片工作台平台化。
- Project / Session / VersionNode 是否需要服务端同步，还是先保持 IndexedDB。
- 搜索 / 命令面板第一阶段要覆盖哪些对象。
- 拼图画布是否采用 Konva + react-konva，并先 local-first 验证导出和版本快照。

## 验收方式

每完成一个阶段，至少检查：

- 对应路线图阶段功能是否完成。
- MVP/V1/封板验收清单是否有新增通过项。
- 密钥是否没有出现在 URL、日志、localStorage 或前端长期明文存储。
- 参数校验是否前后端一致。
- 错误提示是否覆盖 400、401、403、429、超时和上游失败。
