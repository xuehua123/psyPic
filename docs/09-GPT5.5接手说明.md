## 固定边界

- PsyPic 是独立生图站，不嵌进 Sub2API 管理后台。
- 产品必须同时做商业创作台和灵感社区；优先赢商业创作效率，社区作为作品沉淀、同款生成和增长入口。
- 除非用户明确指定其他栈，默认使用 Next.js App Router + TypeScript + BFF Route Handlers。
- MVP 使用 Images API：`/v1/images/generations` 和 `/v1/images/edits`。
- `v0.3` 是文生图 alpha，不是公开试用版本。
- `v0.4` 是公开试用 MVP，包含图生图、单参考图、继续编辑和基础历史。
- `v0.5` 才开始做流式 partial preview 和任务队列。
- `v0.8` 开始强化商业模板和 Prompt 助手。
- `v0.9` 建立社区最小发布模型、基础作品详情、基础同款生成和审核入口。
- `v1.0` 必须包含完整灵感社区信息流、作品详情增强和一键同款生成增强。
- 前端不直接持有长期 Sub2API API Key。
- Sub2API 一键导入必须采用短期 ticket / `import_code` + PsyPic BFF 后端代持。
- 导入交换响应不得返回 `api_key`；后续前端只带 `psypic_session` cookie 调 PsyPic API。
- MVP 历史和图片缓存可先放 IndexedDB，但 BFF 仍必须负责 session、密钥代持、参数校验、限流和转发。

## 新会话第一阶段任务

先实现 `v0.1-v0.2`：

1. 创建 PsyPic 项目骨架。
2. 做桌面端三栏创作台和移动端底部面板布局。
3. 实现手动配置 Sub2API Base URL 和 API Key 的开发模式。
4. 实现 PsyPic BFF 的 `/api/images/generations` 代理。
5. 实现 Sub2API 一键导入的服务端契约雏形。
6. 完成密钥不进 URL、不进前端长期明文存储的检查。

再推进 `v0.3-v0.4`：

1. 补齐文生图完整参数和基础历史。
2. 接入 `/v1/images/edits`。
3. 实现单参考图、结果转参考图、继续编辑。
4. 回归 [07-验收清单.md](./07-验收清单.md) 的 MVP 验收项。

## 需要先问用户的事项

- 是否接受默认 Next.js 技术栈，或明确改为 Nuxt、Vue SPA、React SPA 等其他方案。
- ORM 使用 Prisma 还是 Drizzle。
- 对象存储使用 R2、S3 还是 MinIO。
- PsyPic 是否需要独立用户系统，还是完全依赖 Sub2API 导入 session。
- MVP 是否必须部署上线，目标平台是 Vercel、Cloudflare、Render、自有服务器还是其他。
- 手动 API Key 模式是否只用于开发，还是也面向正式用户开放。
- 公开试用阶段是否允许 4K、`moderation=low`、以及单次 `n > 1`。
- 社区发布是否默认审核，是否允许用户隐藏 prompt、参数和参考图。

## 验收方式

每完成一个阶段，至少检查：

- 对应路线图阶段功能是否完成。
- MVP/V1/封板验收清单是否有新增通过项。
- 密钥是否没有出现在 URL、日志、localStorage 或前端长期明文存储。
- 参数校验是否前后端一致。
- 错误提示是否覆盖 400、401、403、429、超时和上游失败。
