# 08. 风险与待确认决策

## 已确定决策

- PsyPic 是独立站。
- MVP 以 Images API 为主。
- 默认模型是 `gpt-image-2`。
- 不把 API Key 放 URL。
- 一键导入使用短期 ticket。
- 优先复用 Sub2API 作为网关和计费系统。

## 需要确认的产品决策

### 是否必须登录 PsyPic

推荐：一键导入后创建 PsyPic session。手动 key 模式允许无账号试用，但能力受限。

取舍：

- 必须登录：安全、历史同步、计费清晰。
- 允许匿名：上手快，但密钥存储和滥用风险更高。

### 历史存储位置

推荐：MVP 本地 IndexedDB，正式版服务端存储。

取舍：

- 本地：开发快、隐私好、跨设备差。
- 服务端：体验完整、可分享、需要对象存储和权限控制。

### 是否开放 4K

推荐：默认管理员关闭，用户组可配置开放。

原因：

- 4K 成本和延迟更高。
- 官方对 2K+ 输出标注为实验性范围。

### 是否开放 `moderation=low`

推荐：默认关闭，管理员按站点策略决定。

原因：

- 内容安全风险更高。
- 需要配合日志、风控和用户协议。

## 技术风险

### 成本估算不准

风险：

`gpt-image-2` 成本和 token、尺寸、quality、输入图、partial image 都有关，Sub2API 当前按图片档位近似计费会产生偏差。

缓解：

- MVP 明确“估算成本”。
- V1 使用响应 `usage` 精算。
- 对 high/4K/partial image 设置管理员开关。

### 长任务超时

风险：

复杂 prompt 可能接近 2 分钟。

缓解：

- 前端使用任务状态，不阻塞页面。
- 后端超时至少 180 秒。
- 支持取消。
- 对 high/4K 提示耗时更长。

### 密钥泄露

风险：

前端明文存储或 URL 传参会泄露 Sub2API key。

缓解：

- 默认后端代持。
- 禁止 URL 明文 key。
- 日志脱敏。
- Cookie 使用 HttpOnly/SameSite/Secure。

### 图片隐私

风险：

用户上传参考图可能包含敏感内容。

缓解：

- 默认私有。
- 明确保留周期。
- 删除任务时删除对象存储文件。
- 分享必须用户主动打开。

### 社区内容与商业素材泄露

风险：

- 用户误把客户素材、产品图、未发布广告或内部 prompt 公开到社区。
- 社区作品可能包含侵权、敏感、低质或导流内容。
- 公开同款生成可能复制商业创意，引发投诉。

缓解：

- 发布默认私有，公开社区必须二次确认。
- Prompt、参数、参考图和原始素材默认不公开，由用户主动勾选。
- 社区必须支持举报、下架、精选、黑名单和管理员审核开关。
- 商业素材发布前提示版权、肖像权、商标和客户保密风险。

### CORS 和代理

风险：

纯前端直连 Sub2API 可能受 CORS 限制，也会暴露 key。

缓解：

- 生产默认使用 PsyPic BFF。
- 仅本地开发允许直连/调试模式。

## Sub2API 改造风险

### 现有 LobeHub 集成不可直接复用

LobeHub 逻辑可以作为模式参考，但建议抽象成通用外部应用 launch/import 能力，否则后续每接一个应用都复制一套。

### Wire / 分层约束

Sub2API 有严格分层和 Wire 依赖注入：

- `service` 不应 import `repository`、`gorm`、`redis`。
- `handler` 不应 import `repository`、`gorm`、`redis`。
- 修改 DI 后需要在 `backend/` 运行 `go generate ./cmd/server`。

### 生产部署约束

不得在生产业务服务器构建 Docker 镜像。生产主机只能拉取预构建镜像、重启服务、健康检查和回滚。

## 上线前必须回答的问题

- PsyPic 是否有自己的用户系统，还是完全依赖 Sub2API session？
- 图片历史默认保存多久？
- 用户删除任务是否物理删除图片？
- 免费用户是否允许 high quality？
- 是否允许匿名用户手动填 key？
- 是否支持公开分享？
- 公开分享是否加水印？
- 社区发布是否默认需要审核？
- 社区是否允许隐藏 prompt、参数和参考图？
- 管理员如何查看图片成本和异常？
- 失败任务是否扣费，以 Sub2API 实际扣费为准还是 PsyPic 本地规则为准？
