# 08. 风险与待确认决策

## 已确定决策

- PsyPic 是独立站。
- MVP 以 Images API 为主。
- 默认模型是 `gpt-image-2`。
- 不把 API Key 放 URL。
- 一键导入使用短期 ticket。
- 优先复用 Sub2API 作为网关和计费系统。

## 需要确认的产品决策

### 是否必须登录 PsyPic

推荐：一键导入后创建 PsyPic session。手动 key 模式允许无账号试用，但能力受限。

取舍：

- 必须登录：安全、历史同步、计费清晰。
- 允许匿名：上手快，但密钥存储和滥用风险更高。

### 历史存储位置

推荐：MVP 本地 IndexedDB，正式版服务端存储。

取舍：

- 本地：开发快、隐私好、跨设备差。
- 服务端：体验完整、可分享、需要对象存储和权限控制。

### 是否开放 4K

推荐：默认管理员关闭，用户组可配置开放。

原因：

- 4K 成本和延迟更高。
- 官方对 2K+ 输出标注为实验性范围。

### 是否开放 `moderation=low`

推荐：默认关闭，管理员按站点策略决定。

原因：

- 内容安全风险更高。
- 需要配合日志、风控和用户协议。

## 技术风险

### 成本估算不准

风险：

`gpt-image-2` 成本和 token、尺寸、quality、输入图、partial image 都有关，Sub2API 当前按图片档位近似计费会产生偏差。

缓解：

- MVP 明确“估算成本”。
- V1 使用响应 `usage` 精算。
- 对 high/4K/partial image 设置管理员开关。

### 长任务超时

风险：

复杂 prompt 可能接近 2 分钟。

缓解：

- 前端使用任务状态，不阻塞页面。
- 后端超时至少 180 秒。
- 支持取消。
- 对 high/4K 提示耗时更长。

### 密钥泄露

风险：

前端明文存储或 URL 传参会泄露 Sub2API key。

缓解：

- 默认后端代持。
- 禁止 URL 明文 key。
- 日志脱敏。
- Cookie 使用 HttpOnly/SameSite/Secure。

### 图片隐私

风险：

用户上传参考图可能包含敏感内容。

缓解：

- 默认私有。
- 明确保留周期。
- 删除任务时删除对象存储文件。
- 分享必须用户主动打开。

### 社区内容与商业素材泄露

风险：

- 用户误把客户素材、产品图、未发布广告或内部 prompt 公开到社区。
- 社区作品可能包含侵权、敏感、低质或导流内容。
- 公开同款生成可能复制商业创意，引发投诉。

缓解：

- 发布默认私有，公开社区必须二次确认。
- Prompt、参数、参考图和原始素材默认不公开，由用户主动勾选。
- 社区必须支持举报、下架、精选、黑名单和管理员审核开关。
- 商业素材发布前提示版权、肖像权、商标和客户保密风险。

### CORS 和代理

风险：

纯前端直连 Sub2API 可能受 CORS 限制，也会暴露 key。

缓解：

- 生产默认使用 PsyPic BFF。
- 仅本地开发允许直连/调试模式。

### 工作台复杂度膨胀

风险：

项目、会话、版本节点、素材、任务、社区作品如果没有统一语言，会让 UI、API 和数据模型各自长一套概念。

缓解：

- 文档统一使用 project / session / version node / asset / job。
- UI 先保持 local-first 工作台体验，再逐步服务端化。
- 新功能必须说明挂在哪个对象上，不能只新增一个孤立面板。

### 版本图和素材存储膨胀

风险：

非破坏性分叉会保留大量旧节点、参考图、mask、partial image 和导出文件，长期使用后存储成本和检索复杂度会上升。

缓解：

- 区分 version node 元数据、长期 ImageAsset、临时 TempAsset 和导出文件。
- partial image 默认短期保留，成功最终图再进入长期素材库。
- 删除项目或会话时先软删元数据，再异步清理对象存储。

### Job Runtime 可观测性不足

风险：

生成、批量、ZIP、社区审核和素材同步如果各自实现 loading，会导致取消、重试、超时和失败排查不可控。

缓解：

- 所有长任务进入统一 job runtime。
- 记录 request id、upstream request id、状态迁移、耗时和失败原因。
- UI 通过 Task Dock 展示任务，而不是只在按钮上显示 loading。

### local-first 到服务端迁移

风险：

ProjectSidebar、版本图和 Prompt 收藏先走 IndexedDB，后续上服务端时可能出现 id、排序、折叠状态和 active 会话迁移问题。

缓解：

- 先让本地模型对齐未来服务端模型字段。
- 服务端化时做一次只读迁移预览和冲突合并策略。
- active 状态仍以 URL / session state 为准，用户手动折叠状态单独迁移。

### Provider 边界漂移

风险：

如果把 Sub2API 字段直接扩散到工作台层，后续接 Responses image tool、背景移除、超分或审核 provider 时会重写大量 UI 和历史模型。

缓解：

- BFF 内部用 provider adapter 统一 generation、utility、storage、queue。
- version node 保存 PsyPic 标准参数快照，再把 provider-specific payload 放进脱敏 metadata。
- 计费和 usage 统一从 Job / Billing 层读取。

### Board Mode 导出风险

风险：

Board Mode 会把多张素材、手绘标注、mask 和文字 brief 合成一张参考图。如果画布使用外链图片、超大图层或未受控的 canvas 状态，可能导致导出失败、页面卡顿、图片隐私泄露或版本节点快照过大。

缓解：

- 所有进入 Board 的图片先转成 PsyPic asset 或同源代理 URL，避免 canvas tainted 后无法导出。
- BoardDocument 只保存 PsyPic 标准 JSON，不保存第三方 canvas 实例。
- 限制画布像素、图层数量、stroke 点数和 snapshot 字节数。
- annotation layer 和 mask layer 明确分离，导出 edits mask 时只使用 mask layer。
- Board export 进入 ImageAsset / BoardExportAsset，和 version node 建立引用，便于清理、恢复和审计。

## Sub2API 改造风险

### 现有 LobeHub 集成不可直接复用

LobeHub 逻辑可以作为模式参考，但建议抽象成通用外部应用 launch/import 能力，否则后续每接一个应用都复制一套。

### Wire / 分层约束

Sub2API 有严格分层和 Wire 依赖注入：

- `service` 不应 import `repository`、`gorm`、`redis`。
- `handler` 不应 import `repository`、`gorm`、`redis`。
- 修改 DI 后需要在 `backend/` 运行 `go generate ./cmd/server`。

### 生产部署约束

不得在生产业务服务器构建 Docker 镜像。生产主机只能拉取预构建镜像、重启服务、健康检查和回滚。

## 上线前必须回答的问题

- PsyPic 是否有自己的用户系统，还是完全依赖 Sub2API session？
- 图片历史默认保存多久？
- 用户删除任务是否物理删除图片？
- 免费用户是否允许 high quality？
- 是否允许匿名用户手动填 key？
- 是否支持公开分享？
- 公开分享是否加水印？
- 社区发布是否默认需要审核？
- 社区是否允许隐藏 prompt、参数和参考图？
- 管理员如何查看图片成本和异常？
- 失败任务是否扣费，以 Sub2API 实际扣费为准还是 PsyPic 本地规则为准？
- Project / Session / VersionNode 何时从 IndexedDB 迁移到服务端？
- 搜索索引第一阶段只做本地筛选，还是直接引入服务端索引？
- 删除项目时是否级联删除资产，还是只隐藏工作台上下文？
- Board Mode 第一版是否限制为固定画布尺寸，而不是无限画布？
- BoardDocument 是否先 local-first，等导出到 edits 和版本快照跑通后再服务端化？
