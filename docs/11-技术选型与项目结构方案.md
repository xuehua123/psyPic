# 11. 技术选型与项目结构方案

## 默认技术栈

除非用户明确指定其他栈，PsyPic 默认使用：

- Frontend / BFF：Next.js App Router + TypeScript。
- 包管理：pnpm。
- UI：Tailwind CSS + shadcn/ui 风格组件 + lucide 图标。
- 画布编辑：Board Mode 第一版推荐 Konva + react-konva。
- 表单：React Hook Form + Zod。
- 数据请求：TanStack Query。
- 本地历史缓存：IndexedDB，建议使用 Dexie。
- 数据库：PostgreSQL。
- ORM：Prisma 或 Drizzle，二选一即可；默认推荐 Prisma，便于快速建模和迁移。
- Redis：存 session、import code、限流、任务锁。
- 对象存储：Cloudflare R2 / S3 / MinIO，接口按 S3 兼容设计。
- 后台任务：MVP 先同步请求；`v0.5` 后引入轻量 job table + worker。
- 部署：先支持单体部署；后续可拆 Web/BFF/Worker。

## 为什么默认 Next.js

- 创作台和社区都需要高质量前端体验。
- BFF 可以和前端同仓，降低密钥代持、图片代理和 session 开发成本。
- Route Handlers 可承载 `/api/images/*`、导入交换、历史、模板和社区 API。
- 后续做灵感社区时可以逐步引入 SSR/ISR，不需要重构应用形态。

不建议 MVP 使用纯前端 SPA 直连 Sub2API，因为生产环境需要 BFF 代持密钥、统一参数校验、限流、错误标准化和审计。

## 模块边界

### UI 层

- 只处理交互、展示和表单状态。
- 不持有长期 Sub2API API Key。
- 不直接调用 Sub2API。
- 调用 PsyPic 自己的 `/api/*`。

### BFF API 层

- 校验 session。
- 查找 Sub2API key binding。
- 校验参数。
- 调用 Sub2API。
- 标准化错误、usage、request id。
- 写入任务、历史、成本和审计记录。

### Service 层

- `ImageService`：文生图、图生图、同款生成。
- `ImportService`：Sub2API ticket/import code 交换。
- `TemplateService`：商业模板。
- `CommunityService`：作品发布、详情、互动、举报。
- `BillingService`：成本估算和实际 usage 记录。
- `ModerationService`：发布审核、下架、风险提示。

### Repository 层

- 只处理数据库读写。
- 不直接调用 Sub2API。
- 不做 UI 或 HTTP 逻辑。

### Workbench 层

图片工作台建议再单独拆一层语义，不要把项目、版本图、任务 runtime 和历史全部塞进通用 UI 目录里。

- `components/creator/studio/`：Project Sidebar、Main Canvas、Inspector、Task Dock、VersionStream。
- `components/creator/board/`：Board Mode、BoardStage、BoardToolbar、BoardLayerList、BoardInspector、BoardExportPanel。
- `lib/creator/`：本地项目、版本图、选择状态、工作台辅助逻辑。
- `lib/creator/board/`：BoardDocument / BoardLayer 类型、序列化、导出、undo/redo 和版本节点关联。
- `server/services/workbench/`：项目、会话、版本节点、任务 runtime 的领域服务。
- `server/repositories/workbench/`：工作台相关持久化读写。

### Provider 边界

生成、工具、存储和队列建议都通过 adapter 暴露，而不是把 Sub2API 字段直穿到上层。

- generation provider：Sub2API / OpenAI Images / Responses image tool。
- utility provider：背景移除、超分、OCR、审核等未来能力。
- storage provider：local / R2 / S3 / MinIO。
- queue provider：in-memory / Redis / DB-backed worker。

上层只认识统一的工作台参数快照、任务状态和资产对象，这样后续替换 provider 时不会把 UI 和版本图一起震掉。

### Board Mode 技术边界

Board Mode 不要把 canvas 库对象直接写进全局工作台模型。建议边界：

- UI 只在 `components/creator/board/` 中使用 `react-konva`。
- 持久化只保存 PsyPic 自己的 `BoardDocument` / `BoardLayer` JSON。
- 导出只产出标准 `ImageAsset` / `BoardExportAsset`，再进入 `/api/images/edits`。
- `VersionNode` 只保存 `board_document_id`、`board_snapshot`、`board_export_asset_id`，不保存 Konva Stage 实例。
- 如果未来替换 Fabric.js 或 tldraw，迁移面应限制在 board 模块内。

## 环境变量

```env
APP_URL=https://psypic.example.com
DATABASE_URL=postgresql://...
REDIS_URL=redis://...
SESSION_SECRET=...
KEY_ENCRYPTION_SECRET=...
SUB2API_DEFAULT_BASE_URL=https://sub2api.example.com/v1
S3_ENDPOINT=...
S3_REGION=auto
S3_BUCKET=psypic
S3_ACCESS_KEY_ID=...
S3_SECRET_ACCESS_KEY=...
MAX_IMAGE_UPLOAD_MB=20
TEMP_ASSET_TTL_HOURS=72
ENABLE_COMMUNITY=false
ENABLE_PUBLIC_PUBLISH=false
PSYPIC_WORKBENCH_PROJECTS_STORE=indexeddb|database
PSYPIC_BOARD_DOCUMENTS_STORE=indexeddb|database
PSYPIC_SEARCH_INDEX_ENABLED=false
```

安全要求：

- `SESSION_SECRET` 和 `KEY_ENCRYPTION_SECRET` 必须不同。
- API Key 不得写入前端环境变量。
- 生产日志必须脱敏 `Authorization`、`api_key`、`psypic_session`。

## 阶段落地策略

### `v0.1-v0.2`

- 项目骨架。
- BFF route handlers。
- 手动配置开发模式。
- Sub2API 一键导入契约。
- session 和 key binding 后端代持。

### `v0.3-v0.4`

- 文生图、图生图、单参考图。
- IndexedDB 基础历史。
- 商业尺寸预设和单目标尺寸生成/下载。
- MVP 验收闭环。

### `v0.5-v0.7`

- 流式 partial preview。
- 任务队列和取消。
- 服务端任务表。
- 服务端历史、素材库、收藏、ZIP。

### `v0.8-v1.0`

- 商业模板和 Prompt 助手。
- 社区最小发布、审核、举报、下架。
- 灵感社区信息流、作品详情、同款生成。
- 批量任务、CSV、多尺寸批量。

### `v1.1+`

- 图片工作台平台化。
- Project / Session / VersionNode 服务端化。
- Job runtime、search / command surface 和 provider 边界完善。
- Board Mode 本地原型：多素材编排、手绘、mask、导出和 version node snapshot。
- BoardDocument / BoardExportAsset 服务端化和搜索索引。

## 部署策略

MVP 推荐单体部署：

```text
Next.js app
  -> Route Handlers as BFF
  -> PostgreSQL
  -> Redis
  -> S3/R2
  -> Sub2API
```

正式版可拆成：

```text
Web/BFF
Worker
PostgreSQL
Redis
Object Storage
Sub2API
```

生产安全：

- 不在生产业务服务器构建 Docker 镜像。
- 生产只拉取预构建镜像、切换 tag、重启服务、健康检查和回滚。
- 发布前记录当前镜像 tag 和回滚命令。

## 开工前确认

- 是否接受默认 Next.js 技术栈。
- 数据库 ORM 使用 Prisma 还是 Drizzle。
- 对象存储使用 R2、S3 还是 MinIO。
- 部署平台是 Vercel、Cloudflare、Render、自有服务器还是其他。
- 是否需要从第一版就接入真实 PostgreSQL/Redis，还是 MVP 可用本地服务。
- 图片工作台的 project / version graph 是否先 local-first 再服务端化。
- 是否要把 search / command surface 作为独立模块，而不是塞进现有历史面板。
- Board Mode 是否接受 Konva + react-konva 作为第一版依赖，并把 tldraw / Excalidraw 保留为后续评估。
